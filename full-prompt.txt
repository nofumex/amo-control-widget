Используй следующий стек:

* Python 3.12
* FastAPI
* SQLAlchemy 2.x async
* Alembic

* httpx для amoCRM и Telegram API
* Pydantic v2
* pytest
* ruff
* mypy, насколько разумно
* structlog или стандартный logging с JSON formatter
* Docker + Docker Compose

Backend должен быть разделен на API-процесс и worker-процесс.

## Worker

Worker должен выполнять:

* периодическую синхронизацию событий amoCRM;
* построение daily report snapshots;
* отправку отчетов в Telegram;
* retry failed deliveries;
* refresh OAuth tokens;
* обработку webhook event inbox.

Можно использовать ARQ/RQ/Celery/APScheduler. Выбери простой и надежный вариант. Предпочтительно: ARQ или RQ + Redis, либо собственный async worker с polling loop и advisory locks. Главное — чтобы было надежно, тестируемо и понятно.

## Widget/frontend

amoCRM widget должен быть отдельным пакетом:

* plain TypeScript или vanilla JS, совместимый с amoCRM widget model;
* `manifest.json`;
* `script.js`;
* `i18n/ru.json`;
* `i18n/en.json`;
* `templates/` если нужно;
* `styles/` если нужно;
* build script, который собирает `widget.zip`.

Не делай тяжелое SPA на React, если это усложняет загрузку в amoCRM. Можно использовать простой TypeScript + DOM rendering + fetch к backend.

# 6. Алгоритм непрерывной работы

Перенеси и покрой тестами алгоритм из `control-agent`.

Смысл:

* На вход идут timestamps активных событий пользователя за день.
* Timestamps сортируются.
* Дубликаты убираются.
* Суммируются только промежутки между соседними событиями, если:

  * gap > 0
  * gap <= max_gap_seconds
* Если gap больше лимита — он не считается рабочим временем.
* max_gap_seconds = `work_session_gap_minutes * 60`.

Добавь тесты:

```python
continuous_work_seconds([0, 60, 600, 1201, 1261], 600) == 60 + 540 + 60
```

Добавь edge cases:

* пустой список;
* одно событие;
* дубликаты;
* unordered timestamps;
* gap ровно равен лимиту;
* gap больше лимита;
* timezone day boundaries.

# 9. OAuth amoCRM/Kommo

Реализуй полный OAuth lifecycle.

Нужны endpoints:

```text
GET  /api/oauth/install
GET  /api/oauth/callback
POST /api/oauth/refresh/{tenant_id}
POST /api/oauth/disconnect/{tenant_id}
```

Поведение:

* callback принимает `code`, `referer`, `from_widget` и другие параметры, если amoCRM их передает;
* определяет base_url/subdomain;
* обменивает authorization code на access/refresh token;
* получает `/api/v4/account`;
* создает/обновляет tenant;
* сохраняет encrypted tokens;
* создает default report config;
* возвращает понятный HTML/JSON ответ.

Важно:

* access token живет ограниченное время;
* refresh token надо обновлять заранее и транзакционно;
* при каждом refresh сохраняется новый refresh token;
* при OAuth ошибках tenant переводится в `oauth_error`.

Сделай AmoClient:

* async httpx;
* automatic bearer auth;
* automatic refresh on 401, если возможно;
* rate limiting не более 7 RPS на интеграцию и с запасом;
* retry на 429/5xx;
* pagination;
* методы:

  * `get_account()`
  * `get_users()`
  * `get_event_types()`
  * `get_events(...)`
  * `get_note(entity_type, entity_id, note_id)`
  * `get_incomplete_tasks_due_to(...)`
  * `get_pipelines()` если нужно для stage filters
  * `get_custom_fields(entity_type)` если нужно для field selection

* положить payload в `event_inbox`;
* вернуть 200/204 быстро;
* обработку делать в worker.

Важно: webhooks не должны быть единственным источником правды. Daily report должен пересчитываться pull-based через `/api/v4/events`.

# 21. Очень важные продуктовые решения

5. Весь company-specific hardcode из старого проекта нужно заменить настройками.
6. Старую SQLite-схему использовать только как inspiration, но новая БД должна быть multi-tenant.
7. Старую бизнес-логику метрик перенести и покрыть тестами.
8. Реализация должна быть пригодной для private MVP и public Marketplace hardening.
